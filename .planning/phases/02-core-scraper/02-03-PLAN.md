<objective>
Wire the scraper together: `merge.mjs` assembles the two whole-file documents from the per-store build results — carrying forward last-known data and freezing per-store timestamps on error (D-04/D-05), serializing cold-start errors honestly (D-06), and pinning Wasgau to `unavailable` (D-03). `index.mjs` is the orchestrator + CLI entry: it captures one `now`, reads the prior snapshot, makes the single fetch, runs the fault-isolated per-store build loop, validates all three documents through the frozen schema, and writes them atomically so the run ALWAYS completes and writes all three files. An end-to-end test drives the whole pipeline offline against the captured fixture.

Purpose: This is where DATA-05 fault isolation, DATA-06 status/timestamps, and DATA-01 "write a conforming current-offers.json" actually land. It composes the Plan 01 pure transforms and the Plan 02 I/O seams; it re-implements no matching, schema, normalization, or fetch logic.
Output: `scraper/merge.mjs`, `scraper/index.mjs`, the `npm run scrape` script in `package.json`, and `test/scraper.merge.test.mjs` + `test/scraper.run.test.mjs`.
</objective>

**Cold-start serialization (RESEARCH A2 / D-06) — CONFIRMED against the schema:** A never-seen errored store is written as `{ store, displayName, status: "no_offer" }` in `current-offers.json` (StoreOfferSchema permits `no_offer` with no offer fields — verified: the superRefine only requires price/currency/pricePerLitre/validFrom/validTo when `status === "offer"`) AND as `{ store, status: "error", lastUpdated }` in `status.json`. It is NOT written as `status:"offer"` (would throw — no price) and NOT as `status:"error"` in current-offers.json (the snapshot convention is no_offer when there is nothing to show). This is an explicit, tested case in Task 1.

**Synthetic first-run per-store lastUpdated (Open Q1) — DECIDED:** On the FIRST-EVER write of a never-seen store (no prior status.json entry to preserve), its per-store `status.json` `lastUpdated` is set to `now.toISOString()` (the run clock) regardless of outcome — there is no older value to freeze. On every subsequent run, per-store `lastUpdated` bumps only on a successful refresh and is frozen (carried from prior) on error/unavailable. Documented in `merge.mjs` and asserted in Task 1.

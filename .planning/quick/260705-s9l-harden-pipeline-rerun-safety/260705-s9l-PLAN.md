# Quick Task 260705-s9l: Harden pipeline against re-run / concurrency failures

## Problem

Scheduled run `28749099899` failed and re-running it also failed — two distinct causes:

1. **Original failure (attempt 1):** `scrape-and-commit` pushed data fine, but
   `actions/deploy-pages@v5` hit a transient GitHub Pages error
   (`Deployment failed, try again later`). The artifact uploaded correctly; Pages
   rejected the deployment ~5s later. A single transient hiccup failed the run.

2. **Re-run failure (attempt 2):** GitHub's "Re-run" replays from the *original*
   head SHA (`5b00b97`), but origin had already advanced to `74e1a79` (pushed by
   attempt 1). The commit step's `git pull --rebase --autostash` then hit an
   unresolvable merge conflict on the regenerated JSON data files
   (`current-offers.json`, `status.json`, `heartbeat.json`) and exited 1. Every
   re-run of this workflow is structurally doomed by this.

## Tasks

### Task 1 — Make the commit/push rebase conflict-safe
- **File:** `.github/workflows/pipeline.yml`
- **Action:** In the "Commit data + heartbeat" step, change
  `git pull --rebase --autostash origin "${GITHUB_REF_NAME}"` to
  `git pull --rebase -X theirs --autostash origin "${GITHUB_REF_NAME}"`.
  During a rebase the replayed commit is "theirs", so conflicts in the
  regenerated data files auto-resolve in favor of this run's fresh scrape. Makes
  the step idempotent and safe when Re-run replays a stale SHA or a concurrent
  run advanced origin.
- **Verify:** `-X theirs` present on the pull line; YAML still valid.
- **Done:** Commit step no longer aborts on data-file rebase conflicts.

### Task 2 — Tolerate a single transient Pages deploy error
- **File:** `.github/workflows/pipeline.yml`
- **Action:** Split the deploy into a first attempt with `continue-on-error: true`
  plus a short wait and a second `actions/deploy-pages@v5` attempt gated on the
  first attempt failing. Coalesce `environment.url` across both step ids. Keep
  first-party actions only (T-04-07 / no third-party actions). A single transient
  "try again later" no longer fails the run; a persistent failure still does.
- **Verify:** Two `deploy-pages@v5` steps, retry gated on
  `steps.deployment.outcome == 'failure'`, retry has no `continue-on-error` so
  real failures still surface.
- **Done:** One transient Pages error is absorbed; the run stays green.

## Out of scope
- Post-rebase push race retry loop (concurrency group already serializes runs).
- Changing GitHub's Re-run-from-stale-SHA behavior (not controllable in-workflow;
  `-X theirs` makes even a stale re-run succeed).

## must_haves
- **truths:** re-running the pipeline no longer hard-fails on data-file rebase
  conflicts; a single transient Pages deploy error is retried once.
- **artifacts:** `.github/workflows/pipeline.yml` (edited).
- **key_links:** `.github/workflows/pipeline.yml` commit + deploy steps.
